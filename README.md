# Meeting Room Booking Platform

Бэкенд-приложение на Java и Spring Boot для автоматизации бронирования переговорных комнат в офисе или коворкинге. Проект разработан с использованием слоистой архитектуры (Layered Architecture) и лучших практик коммерческой Java-разработки.

---

## 🛠 Технологический стек

* **Язык разработки:** Java 25 (совместимо с Java 17+)
* **Фреймворк:** Spring Boot 3.5.x
    * Spring Web (REST API)
    * Spring Data JPA (Слой работы с данными)
    * Spring Validation (Валидация входящих запросов)
* **База данных:** PostgreSQL (основная СУБД) / H2 Database (In-Memory режим)
* **Библиотеки:** Lombok (генерация бойлерплейт-кода через аннотации и паттерн `Builder`)
* **Тестирование:** JUnit 5, Mockito 

---

## 🏗 Архитектура проекта

Приложение разделено на независимые изолированные слои:
1. `model` (Domain Entities) — описание структуры таблиц базы данных и связей между ними.
2. `repository` (Data Access Object) — интерфейсы Spring Data JPA для выполнения SQL/JPQL запросов.
3. `service` (Business Logic) — ядро приложения, где сосредоточена вся бизнес-логика и валидация правил.
4. `controller` (REST API) — эндпоинты для взаимодействия с внешним миром (принимают запросы и возвращают JSON).
5. `exception` (Error Handling) — кастомные исключения и глобальный перехватчик ошибок.

## 🪄 Ключевые технические фишки

### 1. Защита от накладок по времени (Time Overlapping Validation)
В слое репозитория написан кастомный **JPQL-запрос**, который на уровне базы данных проверяет пересечение временных интервалов. Система гарантирует, что ни одна комната не будет забронирована повторно на один и тот же слот времени. Математическая формула условий:
`existing_start < new_end AND existing_end > new_start`

### 2. Оптимизация производительности (Lazy Loading)
Все связи между таблицами (`@ManyToOne`) переведены на стратегию ленивой загрузки `FetchType.LAZY`. Это избавляет приложение от лишних и тяжелых SQL-операций `JOIN` при чтении списков, предотвращая критическую просадку производительности (проблема N+1).

### 3. Глобальная обработка ошибок (Global Exception Handler)
Реализован централизованный перехват исключений с помощью аннотации `@ControllerAdvice`. При возникновении ошибок (например, если комната уже занята или объект не найден), приложение не возвращает системный Stack Trace (ошибка 500), а отдаёт вежливый, понятный и структурированный JSON со статусами `400 Bad Request` или `404 Not Found`.

## 📡 Эндпоинты REST API

### 👥 Пользователи (`/api/users`)
* `POST /api/users` — Создать нового пользователя.

### 🚪 Комнаты (`/api/rooms`)
* `POST /api/rooms` — Добавить новую переговорную комнату.

### 📅 Бронирования (`/api/bookings`)
* `POST /api/bookings` — Создать новое бронирование (с валидацией времени).
* `GET /api/bookings` — Получить список всех бронирований в системе.
* `DELETE /api/bookings/{id}` — Отменить бронирование по его уникальному ID.

## 🚀 Как запустить проект локально

### 1. Подготовка базы данных (PostgreSQL Binaries)
Сервер PostgreSQL должен быть инициализирован и запущен на порту `5432`. Создайте целевую базу данных через командную строку:
```bash
psql -U postgres
CREATE DATABASE meeting_room_db;
\q
```
2. Настройка application.properties
   Убедитесь, что в файле конфигурации прописаны верные доступы к СУБД (пример для локальной базы без пароля в режиме trust):
```bash
Properties
spring.datasource.url=jdbc:postgresql://localhost:5432/meeting_room_db
spring.datasource.username=postgres
spring.datasource.driverClassName=org.postgresql.Driver
spring.jpa.hibernate.ddl-auto=update
```

3. Запуск приложения
   Запустите главный класс MeetingRoomBookingApplication через вашу IDE или соберите проект через Maven:

```bash
mvn clean spring-boot:run
```
После запуска встроенный сервер Tomcat начнет слушать порт 8080. Таблицы внутри PostgreSQL сгенерируются автоматически!
